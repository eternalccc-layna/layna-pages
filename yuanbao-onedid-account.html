01 · Synergy Architecture · 本次核心

接入 OneID 后的产品联动架构

三端以 OneID 为统一主键；用户授权后数据与上下文互通。重点：浏览器与输入法采集的上下文，如何输送给元宝理解与生成。

以 OneID 为统一主键 · 浏览器 & 输入法为上下文采集端，元宝为理解与生成中枢 · 授权网关贯穿数据与上下文互通

浏览器 → 元宝

网页上下文即时求助

用户在浏览器看某篇网页时，授权后当前网页正文 / 圈选内容可作为元宝的实时上下文，直接「就这篇内容问元宝」，无需复制粘贴。

输入法 → 元宝

输入场景智能续写

用户在任意 App 输入时，授权后当前输入内容 / 场景可让元宝理解意图，回流润色、续写、翻译建议到输入法候选区。

元宝 ↔ 全端

一份记忆，跨端一致

三端共享以 OneID 为主键的长期记忆与偏好，用户在浏览器 / 输入法的行为可沉淀，元宝的个性化能力在各端保持一致体验。

03 · Collaboration Framework

面向 OneID 的可落地合作框架

取方案二视角（统一 OneID + 自定义改造），明确 OneID 团队、元宝、浏览器/输入法三方分工，以及需要一起提给 OneID 的关键需求。

CSIG OneID 团队（统一账号提供方）

集中提供统一账号标识与账号能力
  • OneID 统一签发：作为矩阵唯一用户标识，跨产品资产 / 上下文以 OneID 为主键。
  • 身份鉴权 & Token：提供标准认证 API / SDK，颁发与校验登录态。
  • 三方绑定关系：维护微信 / QQ / Apple 等绑定，提供绑定态查询。
  • 统一管理页 & 注销：个人信息、安全、设备、注销服务归口提供。
  • 账号状态与事件：返回 status，并对绑定 / 注销等变更提供通知能力（需求项）。

元宝（上下文消费 · 智能中枢）

汇聚多端上下文，产出智能能力
  • 以 OneID 建登录态：作为 OneID 消费方接入，与其他端平级。
  • 上下文汇聚与理解：消费浏览器 / 输入法授权后的上下文，理解当下意图。
  • 长期记忆 & 个性化：以 OneID 关联长期记忆，跨端一致。
  • 结果回流：把生成结果按授权范围回流各端（输入建议、网页助手等）。

浏览器 / 输入法（上下文采集 · 消费方）

低门槛接入，采集端侧上下文
  • 以 OneID 建登录态：登录 UI 自实现，调用认证 API。
  • 端侧上下文采集：浏览器采集网页上下文、输入法采集输入场景上下文。
  • 授权后上传：经用户分级授权后，将上下文按 OneID 提供给元宝。
  • 消费回流结果：接收元宝回流的智能建议并呈现（候选区 / 侧边栏）。

04 · Compliance Assessment

综合评估 · 隐私 / 国际化 / 法务合规

隐私 · 单独同意
同法律主体，上下文互通仍需授权
矩阵产品同属一个法律主体，但不同 App 间共享账号资产 / 上下文仍属信息共享。依《个人信息保护法》，需用户明确授权、不默认全通，并在隐私政策中明确披露。
隐私 · 分级授权
最小集默认 + 上下文开关确认
① 默认仅同步「登录态」最小集；② 网页 / 输入内容等上下文需用户开关确认；③ 设置中提供「已授权来源」+ 一键撤销通道。
法务 · 注销联动
注销单个 App ≠ 全矩阵注销
推荐支持注销单个 App 的用户数据，而非默认全矩阵注销。用户注销时询问三选一：一并注销 / 仅断开关联 / 保留独立账号；单 App 注销后跨 App 数据的同步范围需明确约定。
法务 · 上下文留存
上下文按授权范围留存，可追溯
浏览器 / 输入法采集的上下文属敏感信息，需明确留存时效、使用边界与超范围禁用；撤销授权后应停止使用并按约定清理，全链路可追溯。
国际化 · 出海周期
出海合规由海外法务专员负责
出海对外语材料要求较高，从立项到正式出海周期可能约 1 年，需先确定目标国家/地区，可先走中国港澳台再拓展其他地区；出海合规也可能反向影响国内产品规划。
国际化 · 账号体系
出海版建议独立账号体系
GDPR / CCPA 等对账号互通要求更严。出海版本建议拆出独立账号体系、不与国内直接打通，作为国际化框架下的独立分支演进。
