元宝是主脑，矩阵六端是手脚和感官，资产中转站（Asset Hub）是连接器。

项目 元宝 v0.9.3.2 主笔 Layna Wang 更新 2026-06-04 阅读 约 30 分钟
TL;DR · 一页读懂
用元宝主脑 + 多产品矩阵 + 统一资产底座，
把用户跨设备、跨产品的 AI 体验串成一条可流动、可调度、可沉淀的链路。

我们做什么 —— 以元宝 Chatbox「主脑」+ 浏览器/输入法/翻译/录音笔/文件管家/日历 多产品矩阵，打造"上行送上下文、下行被调用"的双向产品集合，底层用统一 Asset Hub 承载用户全生命周期数字资产。

为谁做 —— 跨设备、跨产品的 AI 工具用户。痛点是"在 A 端做的事到 B 端要重做""AI 不知道我之前说过什么""主脑无法调用手脚""各产品资产缺乏流动通道"。

为什么做成 —— 主脑型 AI（元宝）+ 真实用户量级的产品矩阵（QQ 浏览器 35% 登录率实测、搜狗输入法亿级 DAU 等）。

📎 ChatGPT Connectors 是什么？为什么我们要比它多做一步？

是什么 —— OpenAI 在 2025 年推出的官方连接器，把 ChatGPT 接到 Google Drive / Gmail / Calendar / GitHub / Notion / Slack / SharePoint / OneDrive 等第三方系统上，让 ChatGPT 能读取这些系统里的文件、邮件、日程、代码、文档作为对话上下文。授权模型是用户用 OAuth 主动授权某个 Connector，ChatGPT 在对话中按需检索。 —— 同样的事 OpenAI 依赖生态，元宝可以在矩阵产品中闭环。

1.2 真实痛点（精选 · 强相关「账号统一 + 资产底座 + 多端 AI 联动」）

P-1
跨渠道信息散落，AI 不知道我有什么
→ 资产关系图 Asset Graph 解决
「比如收集 50 多个人……有些可能会发错邮箱，有些可能发到微信上，就会导致负责统计的这个人会比较难来统计。」—— 张志轩（大学生）
「不只是我要给他一个文件……参考所有的和项目有关的一些会议、微信群、邮件沟通记录，写出一个情况说明。」—— 陈思雨（金融分析师）
共性 —— 用户的内容散在邮件/群聊/文档/录音多源，每次问 AI 都要重新喂一遍。账号统一 + 资产关系图把这些"我的东西"自动归一，AI 才能"知道我有什么"。
P-2
多渠道任务 / DDL 无统一看板
→ 主脑 + 资产关系图 + 日历端协同
「我可能会就想要他帮我整理一些我的 DDL……他帮我汇总一些信息让我比较清晰的去，不用就是受到这些错过时间、上传错地方、很混乱的干扰。」—— 张志轩（大学生）
「我更希望他能把我收到的邮件……在这边有个小窗口，相当于一个消息栏一样会告诉我，今天的这些待办事项。」—— 陈佳淇（行政）
共性 —— 任务源散在邮件/审批/IM/日历，单端各看各的。账号统一让六端任务汇入同一关系图，主脑生成跨源看板，日历端落地提醒。
P-3
项目审批 / 外部状态变化无主动感知
→ 主脑监测 + 端原生弹窗回写
「他现在没弹窗了……现在你要自己点进去看的就是你对你那批款有没有下来的这个。」—— 温学添（会计）
「帮我记录一下……比方说今天这款哪个领导有没有帮我批下来了，就是签字通过了……帮我录屏记录一下弹窗给我。」—— 温学添（会计）
共性 —— 审批/工单/政务平台状态变化只能轮询自查。账号打通后，浏览器端代为监测，状态变化主脑判定优先级，端原生通道（输入法/日历/浏览器）回写提醒。
P-4
文件分类归档与"所思即所得"检索
→ 资产中转站 Asset Hub + 资产关系图
「它其实放在这个 A 文件夹，但是你自己又觉得它可能在 B 文件夹，在 B 文件夹里面找好久都找不到。」—— 陈佳淇（行政）
「我很少在这个地方找（指搜索框）……我每次就是想打一些关键词，上面我每次都搜不到。」—— 张志轩（大学生）；「我不记得他什么名字，但是我能知道他有什么这个型号的商品……」—— 陈佳淇（行政）
共性 —— 文件按层级文件夹找不到，关键词搜不到。资产中转站记原件，资产关系图记"谁、什么时候、为什么用过"，做语义召回。
P-5
文件去重、版本清理与跨端同步
→ Asset Hub 版本中枢 + L7 跨端协同
「我希望它有个好处就是可以帮我甄别一些，就是疑似有重复类……」—— 温学添（会计）
「我写文章……中间我可能产生了七八个版本……那我到最后这个东西搞完了之后，我可能就把最后一个版本留下来，前面的版本就全部清掉。」—— 苏秦（研究生）
「PDF 版本的，然后它转换成像 Word 的时候，它就会容易出现一些乱码的情况。」—— 程柠（律师助理）
共性 —— 同一份资产多版本散在多端，重复占空间且找不准。账号统一 + 资产中转站做版本聚合与去重，跨端协议保格式一致。
P-6
跨端接力断裂：手机拍/录/截，PC 找不到、用不上
→ 账号统一 + L7 跨端协同基础设施
「我觉得现在网盘有一个不太好的情况……你又得充会员……那我更宁愿我随身带一个 U 盘。」—— 陈佳淇（行政）
「这都是放那块放在浏览器里面去刷时长……」（端外内容如何回流给 AI 处理）—— 阿文（教师）
共性 —— 手机端产出（拍照/录音/截图/聊天链接）回不到 PC 端 AI，跨端接力靠 U 盘和网盘。账号统一是先决条件，资产中转站做接力中枢。
P-7
AI 拿了我什么 / 怎么撤回
→ L3 同意中心 + 资产可见可撤
[用研·间接信号] 受访者多次提及"不敢把敏感材料交给 AI"「公司协议」「身份证号、银行卡号」等场景的犹豫。
共性 —— 跨端资产沉淀的前置条件是用户对"AI 看了什么、存了多久、怎么删"有可控感。这是 P-1~P-6 的合规底座，单独成痛点。

4.4.4 待评审决策的关键技术要点 ⚠️

⚠️ 【账号体系决策点】
方案一 —— 新产品走 QBID 体系 + 非 QB 体系老产品走 APP ID 转 open ID + QB 维持现状。
　优点：复用当前服务集群，开发成本最低
　缺点：部分产品需要打通，如果未来可能拆分独立，账号体系比较难分割
方案二 —— 独立部署新 ID 体系，打通需要 APP ID 转 open ID 体系。
　缺点：开发&维护成本高，20+ 个服务需要重新部署，所有产品需要对齐互通
　优点：独立部署账号安全评估更独立
  • 规范登录字段/画像信息 —— 登录时间、地点、Q36、GUID（老 push）、QBID、QUA2、USkey（安全 ID）
  • 主体问题 —— 打通需要 APP ID 转 open ID 体系前提是同一主体（公司），搜狗输入法不行，需要单独评估流程 —— 尽量新产品都属于腾讯主体
  • 输入法账号体系不仅包含社交账号，还有其他私有账号（邮箱、平台账号密码等） —— 这种需要提供「用户主动绑定社交账号」的产品能力
  • 元宝账号体系调研（是否存在未绑定社交账号的情况，如手机号直接登录、Apple ID 登录） —— 构建虚拟 ID，通过手机号映射跨 APP 资产互通
  • 新产品账号票据时间尽量统一，鉴权机制统一走 QB 服务
  • 终端评估登录 SDK 组件化工作量
#议题已识别约束待评审方向
🔴 1云盘内容（COS）接口可达性云盘资产涉及账号鉴权、付费状态校验、未登录用户处理；既往跨团队拉取尝试均失败；COS 是否愿意开放接口存在不确定性首期降级方案：暂不接入云盘全量文件，优先打通历史记录/书签等轻量级资产；通过「用户主动授权」路径绕开服务端鉴权，但需对方团队承接
🟡 2未登录用户的本地内容文件管家未登录时本地文件不上云，这部分资产服务端无法获取首期收敛：本地内容不进 Asset Hub，待登录场景成熟后再补；评估端侧轻量索引方案的必要性
🟡 3资产存储周期与成本控制全量操作日志预估月成本千万级；冷数据无限期存储不可持续沿用历史记录 30 天保留策略作为参考基线；非关键行为采用「日志上报生成画像」替代「全量资产入库」
🟡 4资产 Schema 细化颗粒度当前 14 类仅为框架级盘点，字段定义、打标规则未对齐；业务方需求边界模糊首期由平台方先圈定核心字段（如历史记录字段集），与业务方按"先框架后细节"节奏对齐
🟡 5跨产品资产调用的合规闭环涉及用户历史等敏感数据，跨产品拉取需校验登录票据 + 审批 + 审计接入层引入密钥协商；敏感资产访问全链路审计可追溯（对齐 5.2 信任锚点）
🌟 决策建议 —— 方案一为推荐方向 —— 以"复用 QB 稳定账号体系 + 老产品通过 open ID 打通"作为最低成本起点，把工程资源集中在资产层（中转站）与协同层（跨端 Handoff），账号层不重复造轮子。后续若个别产品确需拆分独立，再走专项独立部署评估。
