SECTION 01

从对话到长期陪伴。
记忆系统是Agent 的海马体。

把零散的、瞬时的、跨会话的用户痕迹，沉淀成可检索、可演化、可调用的长期记忆，让 AI 不再"金鱼"。

没有记忆系统时

用户的真实抱怨

  • 上周告诉 AI"我吃素"，这周还在推荐红烧肉
  • 输入法输了 5 次"南京游玩"，浏览器搜索零感知
  • 收藏了 30 篇文章，一个月后翻不出来、AI 不主动用
  • 会议笔记改了 3 版，旧版被覆盖，决策因果链丢了

本质：AI 看似聪明，但每次启动都是"白板"，长期使用价值持续衰减。

三步原理

写入 → 加工 → 调用

步骤关键挑战
写入写得快、写得轻、不阻塞主链路
加工新旧冲突如何处理，不能简单覆盖
调用不止靠向量，要带演化历史

三步看似朴素，行业常见方案仅做到了写和读，没做好"演化"。

但行业方案普遍存在三个隐疾——
长期使用之后，才会暴露。

隐疾 1

覆盖式遗忘

改了就丢，没历史版本。用户半年内对某事的态度变了 4 次，AI 还在推第一次否决的方案。

隐疾 2

堆积式噪声

新旧记忆打架，没合并。越用越乱，记忆有效性随时长持续衰减。

隐疾 3

相似度错位

只靠向量检索。关键决策历史压根召回不到。

⚠️ 这三个隐疾，是矩阵长期演进过程中需要提前规避的风险点。 底座规划期识别出来，比上线后再补救成本低一个数量级。

SECTION 02

底座之所以离不开记忆系统，
是因为矩阵协同的终态不是产品互链，而是共脑。

多产品矩阵的资产自然流动 = 输入法 / 翻译 / 录音笔 / 浏览器 / 文件管家 / 日历 六端的用户数据，
能否被 同一份记忆消费、并不断演化？

当前状态

"伪流动"——形似而神不似

现状后果
各端各存数据孤岛，协同口号大于实质
缺统一画像A 产品的偏好 B 产品不知道
资产即文件存得住，用不起来
没有演化偏好被覆盖，矩阵越用越笨
目标状态

"真流动"——记忆系统加持后

  • 业务间默认隔离 · 保护各产品独立性与隐私边界
  • 按需跨业务共享 · 用户授权下，记忆在矩阵内流动
  • 统一演化 · 所有端写入的记忆，演化链统一管理

SECTION 02 · 用户故事

记忆系统带来的 Aha Moment，
是六端资产自然串起来。

不是硬把产品互相调用，而是同一份记忆在背后默默工作——用户感觉到的是「这个矩阵真的懂我」。

Story 01 · 一次南京周末旅行

一次普通的输入，触发了三端默契。

小王周三晚上想去南京玩。他没刻意做规划，只是用了几个产品。

输入法

给老婆发微信打了三次"南京"——拼音候选浮起、聊天里聊到"周末"和"夫子庙"。

日历

周六周日两天没有日程标记 — 系统记下：周末有空 + 高频词南京。

浏览器

第二天打开浏览器，主页直接弹"想要整理一份南京周末攻略吗？" — 提取了昨天的三次输入痕迹。

文件管家

主动浮出去年九月那张南京美食推荐截图，"您之前存过这个，要不要一起看？"

翻译

晚上小王翻译几个景点英文介绍，自动收进同一个"南京 Memo"。

Aha Moment：整个过程小王没有 "整理"、没有跨产品操作、没有点击任何"导出/同步"按钮。 他只是在正常使用，但矩阵已经把跨五端的痕迹凝聚成一份完整的旅行记忆。 这就是「同一份记忆被多个产品消费」的力量。
Story 02 · 一场被自动整理的会议

会开完了，资产已经在等他。

产品经理小李做完一场需求评审会，回到工位甚至还没动手。

日历

当天 10:00–11:30 标记的"需求评审会"刚刚结束。

录音笔

1.5 小时录音自动转录成纪要，关联到日历这个会。

文件管家

会上提到的"上次的设计稿"——文件管家从历史记录里把那张图主动挂回来。

浏览器

会上聊到的几个友商案例链接，浏览器历史里抓出来挂在纪要旁。

输入法

下次小李打字写 PRD 时，关于这个项目的关键术语自动浮出——记忆里 L2 原子事实在工作。

Aha Moment：过去整理一场会议要用 4 个工具来回切换 30 分钟。 现在他什么都没做，纪要、设计稿、参考链接、术语库已经按"同一场会"的语义自然汇拢——这就是 L4 会话摘要 + 演化链在背后跑。
Story 03 · 半年后还认识我的 AI

不是因为它聪明，是因为它记得。

三月份小张说过"我最近在戒甜"。半年后他打开元宝主脑，问"附近有什么好吃的"。

三月

翻译产品里和外国朋友聊天，他打过一句"I'm cutting sugar lately"——L2 写入"用户偏好：低糖"。

五月

录音笔记下他和医生的体检对话提到"血糖偏高"——L2 升级为 L3 画像"健康关注：血糖管理"。

八月

浏览器里他多次搜索"无糖 / 代糖"相关内容——L5 心智沉淀"决策习惯：饮食偏低 GI"。

现在

主脑回答："给你筛了 3 家附近的，全部标了含糖量信息，自动排除了甜品店。"

Aha Moment：小张从来没有主动告诉主脑"我在戒糖"—— 但矩阵通过半年内三个不同产品的痕迹，自然演化出了他的健康画像。 演化链让它知道这是一个持续的偏好，不是一次性的随口说说。

差异化 ①

演化链 · supersedes

新旧冲突时不覆盖、不删除，加指针指向旧节点 → 形成单向链。检索命中链上任意节点 → 自动展开整条链。

案例：完整追踪用户半年内对内容发行态度的 4 次转折，避免推荐已被否决的方案。

差异化 ②

System 1 / System 2 双系统

System 1：实时直觉，用户操作瞬间响应；System 2：异步反思，后台慢做归类、合并、画像。

主链路不阻塞，用户感知不卡，但记忆在后台越用越懂。

SECTION 04

我们的方案是引擎层接入 + 产品层自研。

Hy-Memory 解决了引擎，
我们解决产品形态。

4.1 矩阵接入档位

不是一刀切，而是按场景分层接入

产品档位理由
输入法Lite数据量大、写入侧为主，不需深度个性化
翻译Lite偏功能侧，画像价值低
录音笔Lite → Pro起步轻，待会议摘要场景成熟升 Pro
浏览器（资产底座主消费侧）Pro需要语义合并 + 画像，矩阵主力消费侧
文件管家Lite偏存储，但需提取属性事实接入 L2
日历Lite → Pro时序场景，演化链价值大
元宝主脑 ChatboxUltra入口 + 大脑，必须越用越懂
4.2 资产分层升级

5 类资产 → 对齐 L1–L6

Hy 层Hy 内容我方现状升级动作
L1 原始痕迹原始对话已有 浏览历史、Chat 原文接入 Hy 写入 API
L2 原子事实可检索事实部分 笔记/文件元数据加结构化抽取
L3 身份画像长期偏好缺必补：跨端画像统一收口
L4 会话摘要长会话精华已有 笔记摘要升级为 Hy L4 格式
L5 心智模型决策习惯缺必补：从行为日志抽取
L6 前瞻意图下一步预测缺必补：驱动主动推荐

结论：5 类资产仅覆盖 L1+L4，画像 / 心智 / 前瞻是空白——这就是为什么资产「存得住但流不动」，没有 L3-L6，资产就只是一座静态仓库。

4.3 不可让渡的差异化

Hy 不解决的，必须我们自己做

模块为什么 Hy 不解决我方必须做
资产可见 / 可编辑Hy 是后台引擎黑盒前台资产 UI、5 类资产卡片
跨设备同步规则Hy 管语义不管协议本地 5W + 云端 1000 + 30 天
资产间跳转图Hy 图谱内部用笔记 ↔ 原文 ↔ 对话出处 三跳 UI
用户掌控 / 隐私感Hy 不暴露删除二次确认、关闭弹窗、状态跟随
演化链可视化Hy 内部用 supersedes翻译成 C 端「资产时间线」
